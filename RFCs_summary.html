Introduction

They are the RFCs that define the IRC protocol.
In this summary, we will only cover the most important parts of the RFCs
that are relevant to the implementation of our own IRC server.
That means that everything related to the client side of the protocol
and to the server-to-server communication will be voluntarily ignored.

Note that we are using the Augmented Backus-Naur form to represent
the grammar rules that apply to the different components of the IRC protocol.
However, we also provided a more human-readable description for each rule.

Server

Name

Version

Users

Nickname

Nicknames are the names used by the users on the IRC network
to identify themselves and each other, for example when you want
to send a private message to someone, or to invite someone to a channel, etc...

Hostname

Hostnames are the names of the hosts that the users are running on.
For example, if a user is running on a computer named "mycomputer",
then their hostname will be "mycomputer".

Username

Usernames are the names of the users on their host.
For example, if a user is logged as "John" on their computer,
then their username will be "John".

Realname

Realnames are the real names of the users.
For example, if a user's real name is "John Doe",
then their realname will be "John Doe".

Modes

User modes are flags that indicate different states about the user.
In messages sent and received by the server, user modes are represented by a single letter.

Channels

Name

Topic

Topics are a short description about the channel. Most of the time,
it describes the main subject that is discussed in the channel.

Note that the topic may be an empty string,
in which case it is assumed that the channel has no topic.

Modes

Channel modes are flags that indicate different states about the channel.
Unlike the user modes, some channel modes may be related to some users,
and not only to the channel itself. Therefore, channels may need to have extra fields
to keep track of the users that are related to these modes (e.g. a list of channel operators)
In messages sent and received by the server, channel modes are represented by a single letter.

Parameters

Some channel modes require some parameters. Among the channel modes listed above,
there are those that require some parameters:

  • +I : takes 1 nickname to add to the invite-list of a channel
  • -I : takes 1 nickname to remove from the invite-list of a channel
  • +b : takes 1 nickname to add to the ban-list of a channel
  • -b : takes 1 nickname to remove from the ban-list of a channel
  • +k : takes 1 key to set for a channel, rejecting new users to join it
    unless they provide the correct key.
  • +l : takes 1 number to set the maximum number of users present in a channel
  • +o : takes 1 nickname to add to the chanop-list of a channel
  • -o : takes 1 nickname to remove from the chanop-list of a channel

Messages

Prefix

The purpose of the prefix is to explicitly indicate the true origin of a received message.
However, the prefix is optional, and thus, if a received message has no prefix,
it is assumed that its true origin is the connection from which it was received.

A prefix may be invalid for 3 different reasons, and in each case,
the received message that contain this invalid prefix must be discarded,
and an error must be sent back to the client from which the message was received.

Parameter list

Commands

Commands are the main way to communicate with the server.
They are used to perform several actions such as sending messages,
joining channels, changing nicknames, etc...

There is a list of the different commands that we must implement below.

PASS

parameters = password
1 password

The PASS command can be used only during the registration process.
It is an optional command, depending on whether the server
that the client tries to connect to is password-protected or not.
The PASS command can only be sent once per connection.

When the server receives a PASS message, it must:

  1. check that the connection of the client isn't already authenticated
  2. check that the client provided enough arguments
  3. check that the given password matches the configured one
  4. mark the connection of the client as authenticated
Note: for each assertion that can fail, there is how the server must react if they do:
  • The connection of the client is already authenticated
    ↳ send back an ERR_ALREADYREGISTERED to the client
  • The password argument is missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
     and then close the connection with the client.
  • The given password doesn't match the configured one
    ↳ send back an ERR_PASSWORDMISMATCH to the client
     and then close the connection with the client.

NICK

parameters = [ colon ] nick
  1. Start with 0 or 1 colon
  2. Contain 1 nick

The NICK command is used to change the nickname of a user.

When the server receives a NICK message, it must:

  1. check that the client provided enough arguments
  2. check that the given nickname is valid and not already in use
  3. change the nickname of the user to the given one
  4. send back a NICK message to the client
    to confirm that their nickname has been changed
Note: for each assertion that can fail, there is how the server must react if they do:
  • The nickname argument is missing
    ↳ send back an ERR_NONICKNAMEGIVEN to the client
  • The given nickname is invalid
    ↳ send back an ERR_ERRONEUSNICKNAME to the client
  • The given nickname is already in use
    ↳ send back an ERR_NICKNAMEINUSE to the client

USER

parameters = user space umode-mask space host space realname
  1. Start with 1 user
  2. Contain 1 space
  3. Contain 1 umode-mask
  4. Contain 1 space
  5. Contain 1 host
  6. Contain 1 space
  7. Contain 1 realname

The umode-mask component is a numeric value
that represents the initial user modes of the user.

This numeric value is actually a bit mask,
where each bit represents a specific user mode.
However, only 2 of them are able to be set by the USER command:

  • the bit at the index 2, which represents the `i` user mode
  • the bit at the index 3, which represents the `w` user mode
For example, if the umode-mask component is "8",
8 being 00001000 in binary, then the `w` user mode will be set,
while the `i` user mode will not.

The USER command must be used only during the registration process.
It must be sent only once per connection.

When the server receives a USER message, it must:

  1. check that the client connection is authenticated
  2. check that the client provided enough arguments
  3. check that the given arguments are valid
  4. set the user name, the modes, the host name,
    and the real name of the user to the given ones
  5. send back a RPL_WELCOME to the client,
    to indicate them that they are now registered on the server
  6. send back a RPL_YOURHOST to the client,
    to indicate them the server's host name and version
  7. send back a RPL_CREATED to the client,
    to indicate them when the server was created
  8. send back a RPL_MYINFO to the client,
    to indicate them the server's host name and version, and the supported modes
Note: for each assertion that can fail, there is how the server must react if they do:
  • The client already sent a USER command before
    ↳ send back an ERR_ALREADYREGISTERED to the client
  • The client connection is not authenticated and/or the nickname of the client isn't set yet
    ↳ close the connection with the client
  • The username argument and/or the umode-mask argument
    and/or the hostname argument and/or the realname argument is/are missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
     and then close the connection with the client.
  • The given user name and/or user mode mask and/or host name is/are invalid
    ↳ close the connection with the client.

OPER

parameters = user space password
  1. Start with 1 user
  2. Contain 1 space
  3. Contain 1 password

The OPER command is used to authenticate a user as a local operator.

When the server receives an OPER message, it must:

  1. check that the client provided enough arguments
  2. check that the given username is valid
  3. check that the host of the client
    matches any of the configured ones
  4. check that the given username-password association
    matches any of the configured ones
  5. set the `o` mode for the user
  6. send back a RPL_YOUREOPER to the client,
    to indicate them that they are now an IRC operator
  7. send back a MODE message to the client,
    to indicate them their new modes
Note: for each assertion that can fail, there is how the server must react if they do:
  • The username parameter and/or the password parameter is/are empty
    ↳ send back an ERR_NEEDMOREPARAMS to the client
  • The given username is invalid
    ↳ silently abort the command
  • The host of the client doesn't match any of the configured ones for an IRC operator promotion
    ↳ send back an ERR_NOOPERHOST to the client
  • The given username-password association doesn't match any of the configured ones
    ↳ send back an ERR_PASSWORDMISMATCH to the client

AWAY

parameters = [ custom-away-message ]
0 or 1 custom-away-message

The AWAY command is used to mark/unmark a user as away.
It also allows the user to choose a custom away message.

When the server receives an AWAY message, it must:

  1. determine whether the client provided an argument
    • If they did, then the server must:
      1. set the away message for the user to the given one
      2. set the `a` mode for the user
      3. send back a RPL_NOWAWAY to the client,
        to indicate them that they are now marked as away
    • If they didn't, then the server must:
      1. unset the away message for the client
      2. unset the `a` mode for the user
      3. send back a RPL_UNAWAY to the client,
        to indicate them that they are no longer marked as away

MODE

parameters = nick [ space 1*( ( plus / dash ) 1*umode ) ]
parameters =/ channel *( space ( plus / dash ) *cmode *( space cmode-param ) )
Either 1 of the following:
  • 1 pattern, which must:
    1. Start with 1 nick
    2. Contain 0 or 1 pattern, which must:
      1. Start with 1 space
      2. Contain 1 or more patterns, which must:
        1. Start with either 1 plus or 1 dash
        2. Contain 1 or more umodes
  • 1 pattern, which must:
    1. Start with 1 channel
    2. Contain 0 or more patterns, which must:
      1. Start with 1 space
      2. Contain either 1 plus or 1 dash
      3. Contain 0 or more cmodes
      4. Contain 0 or more patterns, which must:
        1. Start with 1 space
        2. Contain 1 cmode-param

The MODE command is a dual purpose command in IRC.
It is used to modify both user modes and channel modes.

When the server receives a MODE message, it must:

  1. check that the client provided enough arguments
  2. determine whether the command is targeting a user or a channel
    • If the target is a user, then the server must:
      1. check that both the given nickname and the sender nickname are the same
      2. determine whether there are any mode changes to apply
        • If there are, then the server must:
          1. check that the given mode changes are valid
          2. apply the given mode changes to the user
          3. send back a MODE message to the client,
            to confirm that their modes have been changed
        • If there aren't, then the server must
          send back a RPL_UMODEIS to the client,
          to inform them of their current user modes.
    • If the target is a channel, then the server must:
      1. check that the given channel exists
      2. check that the channel supports modes
      3. check that the sender is one of the channel operator
      4. determine whether there are any mode changes to apply
        • If there are, then the server must:
          1. check that the given mode changes are valid
          2. apply the given mode changes to the channel
          3. send a MODE message to every channel member,
            to inform them of the changes
        • If there aren't, then the server must:
          1. send back a RPL_CHANNELMODEIS to the client,
            to inform them of the current channel modes
          2. if the invite-list of the channel contains masks, then the server must:
            1. send back a RPL_INVITELIST to the client for each of them
            2. send back a RPL_ENDOFINVITELIST to the client
            to inform the client of the current invite-list of the channel
          3. if the ban-list of the channel contains masks, then the server must:
            1. send back a RPL_BANLIST to the client for each of them
            2. send back a RPL_ENDOFBANLIST to the client
            to inform the client of the current ban-list of the channel
Note: for each assertion that can fail, there is how the server must react if they do:
  • The nickname/channel argument is missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
  • The given nickname doesn't match the sender nickname
    ↳ send back an ERR_USERSDONTMATCH to the client
  • The given list of user mode changes contains an invalid mode
    ↳ send back an ERR_UMODEUNKNOWNFLAG to the client
  • The given channel doesn't exist
    ↳ send back an ERR_NOSUCHCHANNEL to the client
  • The given channel doesn't support modes
    ↳ send back an ERR_NOCHANMODES to the client
  • The sender is neither a channel operator nor the creator of the channel
    ↳ send back an ERR_CHANOPRIVSNEEDED to the client
  • The given list of channel mode changes contains an invalid mode
    ↳ send back an ERR_UNKNOWNMODE to the client
  • The given list of mode changes contains a mode that requires an argument,
    but the argument is missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
  • The given list of mode changes contains a mode that requires an argument,
    but the argument is invalid
    ↳ silently ignore the mode change and proceed to the next one
  • The given list of modes changes contains a -I with a nickname
    that doesn't match any mask in the invite-list of the channel
    ↳ silently ignore the mode change and proceed to the next one
  • The given list of mode changes contains a +k,
    but the channel has already the `k` mode set
    ↳ send back an ERR_KEYSET to the client
  • The given list of mode changes contains either a +o or a -o
    with a nickname that doesn't match any user in the channel
    ↳ send back an ERR_USERNOTONCHANNEL to the client
  • The given list of mode changes contains a -o with a nickname
    that doesn't match any channel operator in the channel
    ↳ silently ignore the mode change and proceed to the next one

QUIT

parameters = [ quit-message ]
0 or 1 quit-message

The QUIT command is used to disconnect a client from the server.

When the server receives a QUIT message, it must:

  1. send back an ERROR message to the client,
    to confirm that they are about to be disconnected
  2. send a QUIT message to every other client that shares at least 1 common channel
    with the user that is about to disconnect, to inform them of the user's departure
  3. close the connection with the client
  4. remove the user from all their joined channels
  5. remove the client from its list of known clients

Note: for each ERROR and QUIT message sent by the server,
the server must pass in parameter either the given quit message if there is one,
or a default one if there is not.

JOIN

parameters = channel *( comma channel ) [ space key *( comma key ) ] / %x30
Either 1 of the following:
  • 1 pattern, which must:
    1. Start with 1 channel
    2. Contain 0 or more patterns, which must:
      1. Start with 1 comma
      2. Contain 1 channel
    3. Contain 0 or 1 pattern, which must:
      1. Start with 1 space
      2. Contain 1 key
      3. Contain 0 or more patterns, which must:
        1. Start with 1 comma
        2. Contain 1 key
  • 1 octet that has the value 0x30 (0)

The JOIN command is used to make a user join a channel.

When the server receives a JOIN message, it must:

  1. check that the client provided enough arguments
  2. determine whether the first argument is a list of channels to join or the single character 0x30 (0)
    • If it is a list of channels to join, then, for each of them, the server must:
      1. check that the given channel name is valid
      2. check that the user hasn't reached the maximum number of joined channel yet
      3. determine whether the channel corresponding to the given channel name exists
        • If the channel exists, then:
          1. if the channel supports modes, then:
            1. check that the user doesn't match any of the ban masks of the channel
            2. if the channel is marked as invite-only, then check that the user
              either matches one of the invite masks of the channel
              or has been invited by one of the channel operators.
            3. if the channel requires a key to join, then check that the user
              either has been invited by one of the members of the channel
              or matches any of the invite masks of the channel
              or provided the correct key.
          2. check that the channel hasn't reached the maximum number of members yet
        • If the channel doesn't exist, then:
          1. create the channel
          2. if the channel supports modes, then
            add the user to the list of the channel operators
      4. add the user to the list of the channel members
      5. send a JOIN message to every member of the channel,
        to inform them of the user's arrival
      6. send back a RPL_TOPIC to the client,
        to inform them of the topic of the channel
      7. run the NAMES command with the joined channel passed in parameter
    • If it is the single character 0x30 (0), then run the PART command
      for each channel of which the user is a member
Note: for each assertion that can fail, there is how the server must react if they do:
  • The list of channels to join/zero argument is missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
  • A given channel name is invalid
    ↳ send back an ERR_NOSUCHCHANNEL to the client and proceed to the next channel
  • The client has reached the maximum number of joined channels
    ↳ send back an ERR_TOOMANYCHANNELS to the client and proceed to the next channel
  • The client matches one of the ban masks of the channel
    ↳ send back an ERR_BANNEDFROMCHAN to the client and proceed to the next channel
  • The channel to join is marked as invite-only and the client neither matches
    any of the invite masks of the channel nor has been invited by one of the channel operators
    ↳ send back an ERR_INVITEONLYCHAN to the client and proceed to the next channel
  • The channel to join requires a key and the user
    neither matches one of the invite masks of the channel
    nor has been invited by one of the members of the channel
    nor has provided the correct key
    ↳ send back an ERR_BADCHANNELKEY to the client and proceed to the next channel
  • The channel to join has reached the maximum number of members
    ↳ send back an ERR_CHANNELISFULL to the client and proceed to the next channel

PART

parameters = channel *( comma channel ) [ space part-message ]
  1. Start with 1 channel
  2. Contain 0 or more patterns, which must:
    1. Start with 1 comma
    2. Contain 1 channel
  3. Contain 0 or 1 pattern, which must:
    1. Start with 1 space
    2. Contain 1 part-message

The PART command is used to make a user leave a channel.
When the server receives a PART message, it must:

  1. check that the client provided enough arguments
  2. for each given channel name:
    1. check that the channel exists
    2. check that the user is one of the channel members
    3. send a PART message to every member of the channel,
      to inform them of the user's departure
    4. if the user is one of the channel operators,
      then remove them from the list of the channel operators
    5. remove the user from the list of the channel members
Note: for each PART message sent by the server, the server must pass in parameter
either the given part message if there is one, or a default one if there is not.

Note: for each assertion that can fail, there is how the server must react if they do:
  • The list of channels to leave is missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
  • The channel doesn't exist
    ↳ send back an ERR_NOSUCHCHANNEL to the client and proceed to the next channel
  • The user is not a member of the channel
    ↳ send back an ERR_NOTONCHANNEL to the client and proceed to the next channel

TOPIC

parameters = channel [ space colon topic ]
  1. Start with 1 channel
  2. Contain 0 or 1 pattern, which must:
    1. Start with 1 space
    2. Contain 1 colon
    3. Contain 1 topic

The TOPIC command is used to set, unset, or change the topic of a channel.

When the server receives a TOPIC message, it must:

  1. check that the client provided enough arguments
  2. check that the given channel exists
  3. check that the user is one of the channel members
  4. determine whether the client provided a topic to set
    • If they did, then:
      1. if the `t` channel mode is set, check that the user is one of the channel operators
      2. set the topic of the channel to the given one
      3. send a TOPIC message to every channel members,
        to inform them of the new topic of the channel
    • If they didn't, then:
      1. determine whether the current topic of the channel is an empty string
        • If it is, then send back a RPL_NOTOPIC to the client,
          to indicate that the given channel has no topic
        • If it isn't, then send back a RPL_TOPIC to the client,
          to indicate the topic of the given channel
Note: for each assertion that can fail, there is how the server must react if they do:
  • The channel argument is missing
    ↳ send back an ERR_NEEDMOREPARAMS to the client
  • The channel doesn't exist
    ↳ send back an ERR_NOSUCHCHANNEL to the client
  • The user is not a member of the channel
    ↳ send back an ERR_NOTONCHANNEL to the client
  • The user provided a new topic to set and the `t` channel mode is set,
    but the user is not a channel operator
    ↳ send back an ERR_CHANOPRIVSNEEDED to the client

NAMES

parameters = [ channel *( comma channel ) ]
0 or 1 pattern, which must:
  1. Start with 1 channel
  2. Contain 0 or more patterns, which must:
    1. Start with 1 comma
    2. Contain 1 channel

The NAMES command is used to list the users
that are present in 1 or more given channels.
In the absence of parameters, the command must list
the users that are present in every channel the user is in.
When the server receives a NAMES message, it must check
whether a parameter list is given.

  • If there is, then for each given channel, the server must do the following routine:

    Check that the channel exists.
    Then check that the user is in it.
    Then send back a RPL_NAMESREPLY
    to the client, to indicate the users that are present in the channel.

  • If there is not, then for each channel the user is in,
    the server must send back a RPL_NAMESREPLY
    to the client, to indicate the users that are present in the channel.
Finally, the server must send back a RPL_ENDOFNAMES
to the client, to indicate the end of the NAMES list.

Note: for each assertion, if it fails,
then no error reply is sent back to the client,
the process for the current channel is aborted,
and the process for the next given channel starts.

LIST

parameters = [ channel *( comma channel ) ]
0 or 1 pattern, which must:
  1. Start with 1 channel
  2. Contain 0 or more patterns, which must:
    1. Start with 1 comma
    2. Contain 1 channel

The LIST command is used to list the channels and their information.
When the server receives a LIST message, it must check
whether a parameter list is given.

  • If there is, then for each given channel, the server must do the following routine:

    Check that the channel exists.
    Send back a RPL_LIST
    to the client, to indicate the channel and its information.

  • If there is not, then for each existing channel,
    the server must send back a RPL_LIST
    to the client, to indicate the channel and its information.
Finally, the server must send back a RPL_LISTEND
to the client, to indicate the end of the LIST.

Note: for each assertion, if it fails,
then no error reply is sent back to the client,
the process for the current channel is aborted,
and the process for the next given channel starts.

INVITE

parameters = nick space channel
  1. Start with 1 nick
  2. Contain 1 space
  3. Contain 1 channel

The INVITE command is used to invite a user to a channel.
When the server receives an INVITE message, it must check
that the given nickname exists.
Then the server must check that the inviting user
is in the given channel.
Then the server must check that the invited user
is not already in the given channel nor banned from it.
Then the server must check if the given channel is invite-only.
If it is, then the server must check that the inviting user
is a channel operator.
Then the server must check if the invited user is marked as away.
If they are, then the server must send back a RPL_AWAY
to the inviting user, to indicate that the invited user is away.
Else, the server must forward the INVITE message to the invited user,
and finally, send back a RPL_INVITING
to the inviting user, to indicate that the invited user has been invited.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NOSUCHNICK
  • ERR_NOTONCHANNEL
  • ERR_USERONCHANNEL
  • ERR_CHANOPRIVSNEEDED

KICK

parameters = channel *( comma channel ) space kick-target *( comma kick-target ) [ space comment ]
  1. Start with 1 channel
  2. Contain 0 or more patterns, which must:
    1. Start with 1 comma
    2. Contain 1 channel
  3. Contain 1 space
  4. Contain 1 kick-target
  5. Contain 0 or more patterns, which must:
    1. Start with 1 comma
    2. Contain 1 kick-target
  6. Contain 0 or 1 pattern, which must:
    1. Start with 1 space
    2. Contain 1 comment

The KICK command is used to remove a user from a channel.
When the server receives a KICK message, it must first check
that the given parameters fall into one of the following cases:

  • There is 1 channel, and 1 or more targets to kick.
    In that case, the server must check that the channel exists.
    Then the server must check that the kicking user is in the channel
    and is either a channel operator or the creator of the channel.
    Then, for each target, the server must do the following routine:

    Check that the target is present on the channel.
    Then remove the target from the channel,
    forward the KICK message to the target,
    to inform them that they have been kicked,
    and finally send a KICK message to every user
    that are present in the channel,
    to inform them of the target's departure.

  • There are 2 or more channels, and the same number of targets to kick.
    In that case, for each position `p` from first to last,
    the server must do the following routine:

    Check that the channel at position `p` exists.
    Then check that the kicking user is in it
    and is either a channel operator or the creator of the channel.
    Then check that the target at position `p` is present on the channel.
    Then remove the target from the channel,
    forward the KICK message to the target,
    to inform them that they have been kicked,
    and finally send a KICK message to every user
    that are present in the channel,
    to inform them of the target's departure.


Note: for each assertion that can fail, there is how the server must react if they do:
  • ERR_NOSUCHCHANNEL
  • ERR_USERNOTONCHANNEL
  • ERR_NOTONCHANNEL
  • ERR_NEEDMOREPARAMS
  • ERR_CHANOPRIVSNEEDED

PRIVMSG

parameters = msg-target space text-to-be-sent
  1. Start with 1 msg-target
  2. Contain 1 space
  3. Contain 1 text-to-be-sent

The PRIVMSG command is used to send private messages between users
as well as to send messages to channels.
When the server receives a PRIVMSG message, it must first check
that at least 1 target and 1 text to be sent are given.
Then, for each given target, the server must do the following routine:

Determine whether the target is a channel or a user:

  • In case of a channel, then check that it exists.
    Then check whether the `n` mode is set for the channel.
    If it is, then check that the sending user is in the channel.
    Then forward the PRIVMSG message to every user
    that are present in the channel.
  • In case of a user, then check that they exist.
    Then check whether the target user is marked as away.
    If they are, then send back a RPL_AWAY
    to the sending user, to indicate them that the target user is away.
    Else, forward the PRIVMSG message to the target user.


Note: for each assertion that can fail, there is how the server must react if they do:
  • RPL_AWAY
  • ERR_NOSUCHNICK
  • ERR_CANNOTSENDTOCHAN
  • ERR_NORECIPIENT
  • ERR_NOTEXTTOSEND

NOTICE

parameters = msg-target space colon text-to-be-sent
  1. Start with 1 msg-target
  2. Contain 1 space
  3. Contain 1 colon
  4. Contain 1 text-to-be-sent

The NOTICE command is similar to the PRIVMSG command.
The only difference with the PRIVMSG command is that
automatic replies must never be sent in response to a NOTICE message.

MOTD

The MOTD command is used to get the Message Of The Day of the server.
When the server receives a MOTD message, it must check
that there is a configured Message Of The Day to send.
Then, the server must send back a RPL_MOTDSTART
to the client, to indicate them the start of the Message Of The Day.
Then, the server must send back 1 or more RPL_MOTD
to the client, to indicate them the Message Of The Day.
Finally, the server must send back a RPL_ENDOFMOTD
to the client, to indicate them the end of the Message Of The Day.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NOMOTD

VERSION

The VERSION command is used to get the version of the server.
When the server receives a VERSION message,
it must send back a RPL_VERSION to the client,
to indicate them the version of the server.

TIME

The TIME command is used to get the local time of the server.
When the server receives a TIME message,
it must send back a RPL_TIME to the client,
to indicate them the local time of the server.

INFO

The INFO command is used to get the information about the server:
its version, when it was compiled, when it was started,
and other miscellaneaous information which may be considered to be relevant.
When the server receives an INFO message,
it must send back 1 or more RPL_INFO to the client,
to indicate them the information about the server.
Then, the server must send back a RPL_ENDOFINFO to the client,
to indicate them the end of the information about the server.

WHOIS

parameters = nick *( comma nick )
  1. Start with 1 nick
  2. Contain 0 or more patterns, which must:
    1. Start with 1 comma
    2. Contain 1 nick

The WHOIS command is used to get information about a user.
When the server receives a WHOIS message, for each given nickname,
the server must do the following routine:

Check that the given nickname exists, and that its user is not marked as invisible.
Then send back a RPL_WHOISUSER to the client,
to indicate them the user's information.
If the target user is an IRC operator,
then send back a RPL_WHOISOPERATOR to the client,
to indicate them that the target user is an IRC operator.
If the target user is marked as away,
then send back a RPL_AWAY to the client,
to indicate them that the target user is away.
Then send back 0 or more RPL_WHOISCHANNELS to the client,
to indicate them the channels in which the target user is.
Finally, send back a RPL_ENDOFWHOIS to the client,
to indicate them the end of the WHOIS information.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NOSUCHNICK
  • ERR_NONICKNAMEGIVEN

ISON

parameters = nick *( space nick )
  1. Start with 1 nick
  2. Contain 0 or more patterns, which must:
    1. Start with 1 space
    2. Contain 1 nick

The ISON command is used to check whether a nickname is present on the server.
When the server receives an ISON message, it must check that at least 1 nickname is given.
Then the server must send back a RPL_ISON
to the client, with a list of nicknames in parameter,
to indicate them which nicknames are present on the server.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NEEDMOREPARAMS

KILL

parameters = nick space comment
  1. Start with 1 nick
  2. Contain 1 space
  3. Contain 1 comment

The KILL command is used to disconnect a client from the server.
When the server receives a KILL message, it must check
that the killing user is an IRC operator.
Then the server must check that the given nickname exists.
Then the server must forward the KILL message to the target user,
to inform them that they are about to be disconnected.
Then the server must close the connection with the target user,
remove them from every channel they were in, and then
remove them from its list of known clients.
Finally, the server must send a KILL message
to every channel in which the target user was,
to inform them of the user's departure.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NOSUCHNICK
  • ERR_NEEDMOREPARAMS
  • ERR_NOPRIVILEGES

DIE

The DIE command is used to shutdown the server.
When the server receives a DIE message, it must check
that the sending user is an IRC operator.
Then the server must forward the DIE message
to every client that is connected to it,
to inform them that the server is about to shutdown.
Then the server must remove every user from the channels in which they are,
remove them from its list of known clients,
and close the connection with every client that is connected to it.
Finally, the server must shutdown.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NOPRIVILEGES

RESTART

The RESTART command is used to restart the server.
When the server receives a RESTART message, it must check
that the sending user is an IRC operator.
Then the server must forward the RESTART message
to every client that is connected to it,
to inform them that the server is about to restart.
Then the server must remove every user from the channels in which they are,
remove them from its list of known clients,
and close the connection with every client that is connected to it.
Finally, the server must restart.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NOPRIVILEGES

WALLOPS

parameters = text-to-be-sent
1 text-to-be-sent

The WALLOPS command is used to send a message
to all connected users who have the `w` mode set.
When the server receives a WALLOPS message, it must check
that the sending user is an IRC operator.
Then the server must forward the WALLOPS message
to every client that has the `w` mode set,
to inform them of the message.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NEEDMOREPARAMS
  • ERR_NOPRIVILEGES

PING

parameters = colon token
  1. Start with 1 colon
  2. Contain 1 token

The PING command is used to check if a connection is still alive.
When the server receives a PING message, it must check
that a token is given in parameter.
Then the server must send back PONG message
to the client, with that same token in parameter,
to indicate them that the connection is still alive.

Note: for each assertion that can fail, there is how the server must react if they do:

  • ERR_NEEDMOREPARAMS

PONG

parameters = [ server space ] token
  1. Start with 0 or 1 pattern, which must:
    1. Start with 1 server
    2. Contain 1 space
  2. Contain 1 token

The PONG command is used as a reply to a previous PING message.
When the server receives a PONG message, it must check
that a token is given in parameter.
Then the server must check that the given token is the same
as the one given in the previous PING message, sent by the server to the client.
Then the server must update the time of the last activity of the client.

Note: for each assertion that can fail, there is how the server must react if they do:

  • The token argument is missing
    ↳ silently abort the command
  • The client was not pinged before
    ↳ silently abort the command
  • The token argument is missing or is different from the one given in the previous PING message
    ↳ silently abort the command

ERROR

parameters = error-message
1 error-message

Usually, the ERROR command is used to report serious or fatal errors
from a server to another, but in our case, its only use is to inform the client
that the server is about to disconnect them.
Our server is not supposed to receive any ERROR message.
If it happens, it must be silently ignored.

Replies

Most of the messages that the server receives from a client
generate a reply to that client from the server to indicate
whether the operation succeeded or failed.

Every reply must be sent as one message consisting of the sender prefix,
the 3 digit reply code, the target of the reply, and additional arguments if any.

Replies are not allowed to originate from a client.
Any such messages received by the server must be silently discarded.

There is a list of the different replies that we must implement need below.
